## 🐛 Bug 1 — pharmacy dashboard endpoints throw 500

Both endpoints below registered in the dispatch (so the BE team has
controllers in place) but the handler throws on every call:

```
GET /api/v1/pharmacy/customer-prescriptions  → 500 INTERNAL_ERROR
GET /api/v1/pharmacy/consultations           → 500 INTERNAL_ERROR
```

Spec contract these need to match:
[PHARMACY_PUBLIC_API_SPEC.md](./PHARMACY_PUBLIC_API_SPEC.md) §12
(Dashboard API → Prescriptions / Consultations).

FE wire types (already shipped, BE response must match):
[`conddo-app/lib/api/pharmacyDashboard.ts`](../conddo-app/lib/api/pharmacyDashboard.ts).
Key fields BE must return:

**`/pharmacy/customer-prescriptions`** → `CustomerPrescription[]`:
```
id, customerId, customerName, customerPhone?, fileUrl, patientName,
prescriberName, notes?, status (PENDING|APPROVED|REJECTED),
reviewNote?, reviewedAt?, reviewedByName?, submittedAt, orderId?
```

**`/pharmacy/consultations`** → `Consultation[]`:
```
id, customerId?, customerName, whatsappNumber, topic,
preferredTime?, status (PENDING|CONFIRMED|COMPLETED|CANCELLED),
pharmacistNote?, createdAt, completedAt?
```

Also need these write endpoints (FE already has buttons that call them):

```
PATCH /api/v1/pharmacy/customer-prescriptions/{id}/review
  body: { status: "APPROVED" | "REJECTED", reviewNote?: string }

PATCH /api/v1/pharmacy/consultations/{id}/status
  body: { status: ConsultationStatus, pharmacistNote?: string }
```

**Impact if not fixed**: pharmacist can't review customer-submitted
prescriptions or manage consultation requests from the dashboard —
breaks the Seb&Bayor integration. FE currently degrades to empty state
on the 500 so it doesn't crash, but the feature is non-functional.

## 🤔 Product decision needed — orders + bookings tier

`/api/v1/billing/plans` shows the launcher plan with:
```json
{
  "order_management": "false",
  "bookings": "false"
}
```

So `/orders/board` and `/bookings/upcoming` return 403
PLAN_UPGRADE_REQUIRED for the demo tenant (which is on Launcher).

This contradicts the 2026-06-05 spec update in
[conddo-pricing-tiers.md](../conddo-pricing-tiers.md) which moved
both features to Launcher. The FE PlanGate component renders the
upgrade panel cleanly for the 403, so functionally everything works —
but the user experience for a Launcher-tier fashion tenant is:

> Click "Orders" in sidebar → "Order management — Growth plan is
> required, from ₦45,000/month → Upgrade plan"

**Two paths, pick one**:

**(A) Keep as Growth-only** — orders + bookings stay premium. Fashion
tenants on Launcher see the upgrade panel on those routes. No BE work
needed; the FE is already correct.

**(B) Flip to Launcher** — orders + bookings become Launcher features.
**BE: flip two rows in `plan_features` for the launcher plan**:
```sql
UPDATE plan_features
   SET feature_value = 'true'
 WHERE plan_id = (SELECT id FROM subscription_plans WHERE name = 'launcher')
   AND feature_key IN ('order_management', 'bookings');
```
Then update [BILLING_TIERS_SPEC.md §2](./BILLING_TIERS_SPEC.md#2-seed-the-catalog)
feature matrix to reflect.

The pricing-tiers.md spec update I made on 2026-06-05 favoured (B) —
"a tailor on Launcher without order tracking isn't a coherent product."
But product holds the final call.
