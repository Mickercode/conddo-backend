## 0. Global — feature flag gating

Every Beta endpoint must short-circuit when the tenant doesn't have
`tenant_feature_flags WHERE feature_key = '<key>' AND enabled = true`.
Suggested guard, applied per controller class:

```java
@PreAuthorize("@featureFlagGuard.requiresFlag('cashback_loyalty')")
@RestController @RequestMapping("/api/v1/pharmacy/loyalty")
public class PharmacyLoyaltyController { ... }
```

…where `FeatureFlagGuard.requiresFlag(String key)` looks up the row and
returns `false` (→ 403 with a specific error code) when the flag is
absent or `enabled = false`.

**Error envelope FE expects on a feature-gate refusal:**

```json
{
  "success": false,
  "error": {
    "code": "FEATURE_NOT_ENABLED",
    "message": "This feature is not yet enabled on your account.",
    "details": [{ "field": "featureKey", "message": "cashback_loyalty" }]
  }
}
```

The FE `BetaFeatureGate` component reads `GET /api/v1/feature-flags` on
mount so most users never reach a feature-gated endpoint, but the
server-side guard is the source of truth — never skip it.

**Feature keys (must match `tenant_feature_flags.feature_key` exactly):**

| Key | Beta |
|---|---|
| `cashback_loyalty` | 1 |
| `followup_workflow` | 2 |
| `drug_programs` | 3 |
| `emr_basic` | 4 |

## 1. Cashback Loyalty (Beta 1)

### DDL — already in the roadmap doc §Beta 1

`pharmacy_loyalty_config`, `pharmacy_customer_wallets`,
`pharmacy_wallet_transactions` — copy verbatim from
[`conddo-pharmacy-roadmap.md`](./conddo-pharmacy-roadmap.md#beta-1-cashback-loyalty-system).

### Endpoints

| Method | Path | Auth | Notes |
|---|---|---|---|
| `GET` | `/api/v1/pharmacy/loyalty/config` | tenant-staff | 404 if no row → FE shows "Cashback isn't configured yet" empty state |
| `PUT` | `/api/v1/pharmacy/loyalty/config` | tenant-admin | upsert |
| `GET` | `/api/v1/pharmacy/loyalty/wallets?search=` | tenant-staff | search matches customer name / phone |
| `GET` | `/api/v1/pharmacy/loyalty/wallets/{customerId}` | tenant-staff | 404 if no wallet yet |
| `GET` | `/api/v1/pharmacy/loyalty/wallets/{customerId}/transactions` | tenant-staff | timeline for the per-customer modal (not yet on FE; ship anyway) |

### Response shapes

```typescript
// GET /config + PUT /config
LoyaltyConfig {
  cashbackRate: number;     // percent, e.g. 2 = 2%
  minRedemption: number;    // NGN, e.g. 500
  isActive: boolean;
  updatedAt?: string;
}

// PUT body (all optional)
LoyaltyConfigInput = Partial<LoyaltyConfig>

// GET /wallets
CustomerWallet[] = {
  customerId: string;
  customerName?: string | null;   // server-joined; absent on single-wallet
  customerPhone?: string | null;
  balance: number;
  totalEarned: number;
  totalRedeemed: number;
  updatedAt?: string;
}[]

// GET /wallets/{id}/transactions
WalletTransaction[] = {
  id: string;
  transactionType: "CASHBACK_EARNED" | "REDEMPTION" | "ADJUSTMENT" | "EXPIRY";
  amount: number;            // signed: + credit, − debit
  referenceId?: string | null;  // order_id for CASHBACK_EARNED|REDEMPTION
  note?: string | null;
  createdAt: string;
}[]
```

### Crediting + redemption mechanics (BE-side, no FE)

1. **Order DELIVERED transition** — when an order moves to `DELIVERED`,
   if `pharmacy_loyalty_config.is_active` and the order's customer
   exists, credit `order.total * cashbackRate%` to their wallet as
   `CASHBACK_EARNED`. **Never on placement** — wait for DELIVERED.
2. **Checkout redemption** — `POST /public/{slug}/pharmacy/orders` body
   gains an optional `cashbackRedemption: number` field. Validate that
   the customer has ≥ that balance and ≥ `minRedemption`; deduct as
   `REDEMPTION` and apply as an order-level discount. Reject if it
   would zero out the order or go negative.
3. **No FE work for either** — both fully BE-driven. Customer sees
   their balance via the public website (already specced in
   `PHARMACY_PUBLIC_API_SPEC.md`).

### Acceptance

```bash
# Init config
curl -X PUT $BASE/api/v1/pharmacy/loyalty/config -H "$AUTH" \
  -d '{"cashbackRate":2,"minRedemption":500,"isActive":true}'

# After an order is DELIVERED, the customer's wallet should auto-credit
curl $BASE/api/v1/pharmacy/loyalty/wallets/$CUSTOMER_ID -H "$AUTH"
# → balance > 0 with one CASHBACK_EARNED tx
```

### Billing — **Paystack subscription** (not Routepay)

> **Strategy change 2026-06-11:** Routepay handles tenant→customer
> financials only (deposits, bookings, walk-in orders). All **recurring
> subscriptions** — both Conddo's plan billing and tenant-to-customer
> program subscriptions — go through **Paystack**. See §8 below for the
> full split and the Paystack contract.

The FE already redirects on enrol:

1. On `POST /enroll` (manual or customer self), BE creates a Paystack
   transaction with `plan_code` matching this program's plan and
   `customer.email` for the patient. Return
   `{ enrollment, authorizationUrl, reference }`. The FE opens
   `authorizationUrl` in a new tab.
2. Enrollment row starts as `PENDING_PAYMENT` (BE-side status; FE
   already accepts this as a known extended state).
3. Paystack webhook `subscription.create` → flip the row to `ACTIVE`,
   set `paystack_subscription_code`, set `next_billing_at` to first
   month + 1 month.
4. Paystack webhook `charge.success` for renewals → advance
   `next_billing_at` by 1 month; create `pharmacy_program_charges`
   audit row (new table).
5. Paystack webhook `subscription.disable` or payment failure cascade
   → flip to `PAUSED`; notify the pharmacist via the existing in-app
   notifications.
6. Enforce `ends_at` — if `duration_months` is set, compute
   `ends_at = enrolledAt + duration_months` at enrolment; cron flips
   `ACTIVE` to `COMPLETED` when reached (and cancels the Paystack
   subscription so we stop charging).

**Paystack fees:** Paystack charges 1.5% + ₦100 per transaction (capped
at ₦2,000 on local cards). Pass through to the customer or eat — FE
defers to BE's decision. Surface a `feeAmount` on the enrolment if
passing through, or leave the displayed price as the customer-facing
total if eating it; let me know and I'll align the copy.

**Role guard — IMPORTANT.** The roadmap spec calls for `PHARMACIST` and

### Document storage — security

Per roadmap §Beta 4:

- Private MinIO bucket per tenant (or use the existing `tenants/{id}/`
  prefix on the shared bucket — your call).
- Return **pre-signed read URLs** with short expiry (≤ 1 hour) in the
  `fileUrl` field. FE opens them in a new tab; never store the raw S3
  key.
- All EMR endpoints **audit-logged** — write a row to the existing
  audit log for every read, write, and download. Roadmap calls this
  out explicitly.

### Multipart upload contract

The FE sends:

```
POST /api/v1/pharmacy/emr/{customerId}/documents
Content-Type: multipart/form-data

file:    <binary>   (PDF or image, ≤ 20 MB — FE pre-flights)
docType: "LAB_RESULT" | "PRESCRIPTION" | "REFERRAL" | "IMAGING" | "OTHER"
```

Returns `EmrDocument` (201). BE rejects:

- Wrong content-type (only PDF + image/*)
- Files > 20 MB (FE already gates, but defence in depth)

## 8. Payments strategy split — Paystack + Routepay (new 2026-06-11)

Decision: **Paystack handles every recurring subscription on the
platform. Routepay handles tenant→customer one-off financials.**

| Surface | Provider | Why |
|---|---|---|
| Conddo's own plan billing (Launcher / Growth / Scaler) | **Paystack** | Recurring tenant subscriptions. Native Paystack subscriptions API + webhook automation. |
| Pharmacy Drug Programs (customer subscriptions) | **Paystack** | Same recurring model as plan billing. Reuses Paystack infra. |
| Tenant deposits on bookings / orders / sessions | **Routepay** | One-off, tenant-led collection. Existing flow; no change. |
| Walk-in customer POS (pharmacy) | **Routepay** | One-off. Existing flow; no change. |
| Brand Packages monthly bundle | **Paystack** (when you wire it) | Recurring — moves to the new pattern next time it's touched. |

### 8.1 Paystack contract — Conddo plan billing

| Method | Path | FE expects | Notes |
|---|---|---|---|
| `POST` | `/api/v1/billing/checkout` | `{ authorizationUrl, reference, accessCode? }` | Body `{ planId, billingCycle }`. BE creates the Paystack transaction with the tenant + plan metadata and returns the hosted-checkout URL. |
| `GET` | `/api/v1/billing/verify?reference=<ref>` | `{ status: "success"\|"pending"\|"failed"\|"abandoned", reference, amount?, paidAt?, failureReason?, subscription? }` | Polled by `/settings/billing/return`. `subscription` always present on `success`. |
| `GET` | `/api/v1/billing/subscription` | unchanged | Current sub. |
| `GET` | `/api/v1/billing/plans` | unchanged | Plan catalogue. |
| `POST` | `/api/v1/billing/cancel` | unchanged | Cancels Paystack subscription at period end; the FE shows the existing "Cancels on <date>" copy. |

**Callback URL.** Paystack `callback_url` must point at the **tenant's
host**:

```
https://{tenant-host}/settings/billing/return
```

If you're behind the platform-shared FE deployment (one Vercel build
serving every tenant subdomain), the host header tells you which
tenant the callback belongs to.

**Webhook events to handle:**

| Event | Action |
|---|---|
| `charge.success` (first charge — sub creation) | Mark transaction `success`; create/update `billing_subscriptions` row with the new plan, `expires_at = now + cycle`. |
| `subscription.create` | Persist Paystack `subscription_code` on the tenant's row. |
| `subscription.disable` | Flip sub to `cancelled`; keep `expiresAt` until period end. |
| `invoice.payment_failed` | Move sub to `grace`; existing FE already handles the chip. |
| `invoice.create` (renewal) | Surface in tenant notifications. |

The FE return page (`/settings/billing/return`) polls verify every 3s
for 60s then every 10s up to attempt 40. Most webhook deliveries beat
the customer's redirect — verify usually returns `success` on the
first call.

### 8.2 Paystack contract — Pharmacy Drug Programs

| Method | Path | FE expects | Notes |
|---|---|---|---|
| `POST` | `/api/v1/pharmacy/programs/{id}/enroll` | `{ enrollment, authorizationUrl?, reference? }` | Body `{ customerId, returnUrl? }`. When `authorizationUrl` is set, FE opens it in a new tab; enrollment is `PENDING_PAYMENT` until Paystack webhook confirms. |

Webhook flow is symmetric to §8.1 but scoped to the program
subscription — `charge.success` for the first month flips the
enrollment to `ACTIVE`, sets `next_billing_at`. See §3 for the full
lifecycle.

### 8.3 Why split

- **Paystack** has the cleanest subscriptions API in Nigeria; their
  webhook + retry logic handles billing failures, card updates, and
  customer-initiated cancellation without us reinventing the wheel.
- **Routepay** stays for one-off financial flows where their tenant-
  facing UX (their hosted checkout, escrow, settlement timing) wins.
  Tenants already know Routepay through the deposit flow on bookings;
  no need to switch.
- **No double provider in one surface.** A given page either uses
  Paystack or Routepay — never both — so users aren't asked to enter
  card details twice.
